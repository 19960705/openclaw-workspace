## Telegram 架构优化思路 (00:56-00:59)
- **核心概念**: Telegram 消息处理可以分流
  - 不需要本机能力 → ACP Runtime 子线程直接处理
  - 需要本机能力 → 转发到本机 Gateway
- **优势**:
  - 更快（无需本机中转）
  - 更轻量（ACP Runtime 直接处理）
  - 减少本机负载
  - 避免 Gateway 连接问题
- **适用场景**:
  - ✅ 纯文本对话、信息查询、数据分析
  - ❌ 执行本机命令、读写本地文件、操作本机应用
- **实现方式**: 
  - 判断消息类型 → 路由到 ACP Runtime 或本机 Gateway
  - 使用 sessions_spawn 创建 ACP 子线程
- **待实现**: 需要在 Telegram binding 中添加智能路由逻辑


## Codex 三方协作系统理解 (00:59-01:02)
- **核心架构**: 飞书 ↔ OpenClaw ↔ Codex (本地)
- **工作流程**:
  1. Codex 输入 → OpenClaw 捕获 → 转发到飞书
  2. 飞书消息 → OpenClaw 接收 → 推送到 Codex
  3. Codex 执行结果 → OpenClaw 捕获 → 同步到飞书
  4. 所有对话历史 → Codex session 持久化
- **关键特性**:
  - ✅ 双向实时同步
  - ✅ 多端可见（飞书、Telegram、Codex、终端）
  - ✅ Session 持久化（Codex 插件/app 共享）
  - ✅ 终端执行日志同步
- **技能作用**: 
  - codex-session-handoff: 建立 OpenClaw ↔ Codex 双向通道
  - 通过 session handle 连接同一会话
  - OpenClaw 作为消息中转枢纽
- **应用场景**:
  - 在飞书指挥 Codex 写代码
  - 在 Codex 看到飞书指令
  - 在终端看到完整执行过程
- **前提条件**: 需要 Codex CLI 支持 session 管理
